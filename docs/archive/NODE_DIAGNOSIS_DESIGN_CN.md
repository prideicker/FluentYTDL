# 代理节点诊断体系设计文档

## 1. 背景

当前项目已经实现了一个“网络测试 / 风控探针”页面，但现有方案存在明显的假阳性问题：

1. 诊断链路与真实下载链路不一致。
2. 单个 `403` / `LOGIN_REQUIRED` 信号被解释为最终结论。
3. 固定 `android_vr + youtubei/player + 固定视频` 的探针无法代表 FluentYTDL 的真实下载行为。

这导致用户会遇到一种非常差的体验：

- 测试页提示 `403/Bot`
- 但实际下载仍然可能成功
- 或者能开始下载，只是在中途失败

因此，本次设计的目标不是继续增强“独立探针”，而是建立一套**基于真实 yt-dlp 运行参数的、分层的、低误报的代理节点诊断体系**。

---

## 2. 本次设计的关键前提

本设计基于以下结论收敛：

### 2.1 默认模式是当前唯一可靠主路径

根据项目现状、现网日志，以及 yt-dlp issue `#12482` 的现象，当前 YouTube 的变化使得：

- `web` 客户端的部分 HTTPS 格式会因为缺少 GVS PO Token 被跳过
- 但 yt-dlp 默认模式仍可通过其内部客户端选择策略获取可下载格式

在该 issue 的示例中，yt-dlp 在发现 `web client https formats require a GVS PO Token` 后，仍继续走 `tv client config` / `tv player API JSON`，并最终下载 `format 18`。这说明：

- **默认模式仍然是当前最稳的真实运行路径**
- **不应在诊断中人为强行固定单一 player_client 作为主依据**

参考：

- https://github.com/yt-dlp/yt-dlp/issues/12482

### 2.2 Cookie 的职责是“身份解锁”，不是“网络解封”

当前设计明确将 Cookie 的作用限定为：

1. 解锁年龄限制
2. 解锁会员 / 私密 / 登录后可见内容
3. 提供更稳定的账号态访问上下文

但 Cookie **不应被解释为**：

1. 可以绕过 YouTube 的网络层风控
2. 可以消除节点 / IP 的低信任度
3. 可以替代节点质量本身

因此，节点诊断必须把“身份可访问性”和“节点可接受性”分开描述。

### 2.3 PoToken 当前不是主判定依赖

PoToken 在架构上仍有价值，尤其对未来 YouTube 客户端策略变化具有前瞻意义；但基于当前项目运行现实，本设计明确约束：

- **PoToken 不进入节点主诊断结论**
- **PoToken 不作为“节点可用 / 不可用”的关键判据**
- **PoToken 仅作为辅助上下文记录**

换句话说，本次诊断系统要先回答：

> 当前代理节点在 FluentYTDL 的默认模式下是否可接受

而不是：

> 当前节点是否支持某种未来可能成为必需的附加令牌体系

---

## 3. 设计目标

### 3.1 目标

建立一套面向用户、低误报、可解释的节点诊断系统，用于回答：

1. 当前代理是否能连接到 YouTube
2. 当前代理在 FluentYTDL 默认模式下是否可正常解析目标视频
3. 当前代理是否仅对匿名访问不友好，但登录态仍可用
4. 当前代理是否存在媒体下载阶段风险
5. 当前失败更像是身份问题、节点问题，还是下载阶段稳定性问题

### 3.2 非目标

本系统不试图回答以下绝对化问题：

1. “这个节点是否被 YouTube 永久封禁”
2. “这个节点对所有 YouTube 视频都不可用”
3. “只要探针失败，真实下载就一定失败”

本系统输出的是**兼容性评估与风险等级**，而不是法律意义或平台层面的绝对封禁判断。

---

## 4. 核心设计原则

### 4.1 诊断必须复用真实下载栈

所有核心探测必须尽量复用：

1. `YoutubeService.build_ydl_options()`
2. 当前代理配置
3. 当前 Cookie 状态
4. 当前 yt-dlp 可执行文件与运行环境
5. 当前默认 extractor/client 策略

禁止继续以“独立 urllib 探针”的结果直接输出最终节点结论。

### 4.2 分层诊断，不做单信号硬判

任意单个信号都不能直接推导出“节点已被封”：

- `HTTP 403`
- `LOGIN_REQUIRED`
- `Sign in to confirm you're not a bot`
- 匿名解析失败

这些只能构成某一层的风险信号，必须结合其他层共同判定。

### 4.3 优先使用目标 URL，连接池仅作补充

节点诊断应优先针对：

1. 用户当前输入的 URL
2. 最近一次失败任务的 URL
3. 无目标 URL 时才回退到探测连接池

原因很简单：

- 节点兼容性是“目标相关”的
- 用固定公开视频做最终裁决，会放大偶然性

### 4.4 结论必须是风险等级，不是红绿灯式宣判

UI 和日志层统一输出：

- `good`
- `limited`
- `risky`
- `blocked`

其中 `blocked` 只用于：

- 基础网络不可达
- 或相同配置下多次关键层探测全部失败，且证据高度一致

默认不要轻易使用“已被 YouTube 封锁”这种强词。

---

## 5. 总体架构

建议新增统一模块：

- `src/fluentytdl/youtube/node_diagnosis.py`

负责协调如下子能力：

1. 基础网络可达性检测
2. 默认模式轻量解析检测
3. 认证态轻量解析检测
4. 媒体链路短探测
5. 结果聚合与风险判定
6. 缓存、节流、历史记录

建议数据结构：

```python
@dataclass(slots=True)
class NodeDiagnosisCheck:
    name: str
    signal: str
    status: str          # pass / warn / fail / skipped
    detail: str
    latency_ms: int
    profile: str         # default / auth / anonymous / media


@dataclass(slots=True)
class NodeDiagnosisResult:
    level: str           # good / limited / risky / blocked
    confidence: str      # low / medium / high
    summary: str
    suggestions: list[str]
    checks: list[NodeDiagnosisCheck]
    target_url: str | None
    used_probe_pool: bool
```

## 6. 诊断分层

## 6.1 第 0 层：基础连通性

### 目的

判断代理是否能连到 YouTube，而不是判断风控。

### 建议探测对象

1. `https://www.youtube.com/generate_204`
2. `https://www.youtube.com/robots.txt`

### 允许输出的结论

1. 可达
2. 不可达
3. 超时
4. 代理配置异常

### 不允许输出的结论

1. 节点被 YouTube 风控
2. 节点触发 Bot

---

## 6.2 第 1 层：默认模式轻量解析

### 目的

判断当前代理在 **FluentYTDL 默认模式** 下是否能完成解析。

这是整个新设计的主判断层。

### 实现原则

1. 复用 `YoutubeService.build_ydl_options()`
2. 不强制覆写 `player_client`
3. 不人为切到 `android_vr`
4. 允许 yt-dlp 走其默认客户端组合策略

### 建议执行方式

对目标 URL 运行轻量命令：

```text
yt-dlp --dump-json --skip-download --no-warnings --no-playlist <url>
```

但实际参数需由程序统一构造，继承当前：

- proxy
- cookiefile（如果默认路径本身会带）
- js runtime
- 其他必要 extractor_args

### 这一层回答的问题

> 在当前程序默认真实配置下，这个节点是否支持基本解析

如果这一层成功，则说明：

- 该节点至少对当前程序的主工作流是兼容的

这时就不应再用其他“独立探针失败”去覆盖这个结论。

---

## 6.3 第 2 层：匿名能力探测

### 目的

区分“节点本身不友好”和“仅靠身份才能过”。

### 实现原则

基于与第 1 层相同的默认配置，去掉账号态相关上下文：

1. 去掉 `cookiefile`
2. 不把 PoToken 作为必要项
3. 保留 proxy / js runtime / 默认模式

### 这一层回答的问题

> 这个节点在匿名态下是否也可被 YouTube 接受

### 结果解释

如果：

- 第 1 层通过
- 第 2 层失败

则结论应为：

- 节点匿名访问受限，但当前登录态可用

这属于 `limited`，不是 `blocked`。

---

## 6.4 第 3 层：身份解锁能力

### 目的

判断 Cookie 是否确实为目标视频提供了身份解锁价值。

### 何时执行

仅在以下情况下执行：

1. 目标视频疑似有年龄 / 私密 / 登录态限制
2. 匿名层失败，且需要区分是否为身份问题
3. 用户主动执行深度诊断

### 判定方式

比较：

1. 默认模式带 Cookie 的解析结果
2. 匿名模式解析结果

### 结论范围

1. Cookie 提供了身份解锁能力
2. Cookie 未显著改善可访问性

### 注意

这一层只能说明身份价值，**不能说明节点已解封**。

---

## 6.5 第 4 层：媒体链路短探测

### 目的

解决“能解析，但下载到一半 403”的场景。

### 背景

现有日志已经表明存在：

1. 可以开始下载
2. 可以下载到若干 MB
3. 随后出现 `HTTP Error 403`
4. 断点续传继续撞同一问题

因此，仅解析成功不代表下载稳定。

### 实现原则

1. 这层默认不在每次自动诊断中强制执行
2. 仅在以下场景启用：
   - 用户主动点“深度诊断”
   - 最近一次真实下载已出现中途 403
   - UI 需要判断“解析成功但下载仍失败”的原因

### 实现方式

优先选择下列轻量策略之一：

1. `yt-dlp --test`
2. 下载小格式 / 小片段到临时目录后立即清理

### 结论范围

1. 媒体链路可用
2. 媒体链路高风险
3. 稳定性存疑

这里建议使用 `warn/fail`，不要直接给出“节点彻底失效”。

---

## 7. 探测目标策略

## 7.1 优先级

诊断目标 URL 选择顺序如下：

1. 当前用户输入 URL
2. 最近一次失败任务 URL
3. 最近一次成功任务 URL
4. 探测连接池 URL

## 7.2 连接池的定位

探测连接池仍然保留，但用途收敛为：

1. 冷启动诊断
2. 无用户目标 URL 时的基础评估
3. 诊断采样补充

不允许把连接池结果单独用作最终裁决。

## 7.3 连接池建议分类

建议将探测池分层：

1. `public_easy_pool`
2. `public_common_pool`
3. `public_sensitive_pool`

其中：

- `easy` 用于基本可达与轻量解析
- `common` 用于一般兼容性采样
- `sensitive` 仅用于人工或深度诊断，不用于默认自动探测

---

## 8. 风险判定矩阵

## 8.1 Good

条件示例：

1. 基础连通性通过
2. 默认模式解析通过
3. 媒体短探测通过或未执行但近期真实下载成功

输出：

- 节点与当前程序默认模式兼容

## 8.2 Limited

条件示例：

1. 基础连通性通过
2. 默认模式解析通过
3. 匿名探测失败
4. 认证态探测通过

输出：

- 节点对匿名访问不友好，但当前登录态仍可用

## 8.3 Risky

条件示例：

1. 基础连通性通过
2. 默认模式解析失败
3. 匿名与认证态均失败
4. 或解析通过但媒体短探测失败

输出：

- 当前节点对 YouTube 存在明显限制，或媒体链路高风险

## 8.4 Blocked

仅在高置信度条件下使用：

1. 基础网络不可达
2. 或在同目标、同配置、短时间重复验证下关键层全部失败
3. 且错误信号高度一致

输出：

- 当前节点在当前配置下不可用

注意：这里依然建议 UI 用语偏工程化，例如：

- “当前节点在默认模式下不可用”

而不是：

- “已被 YouTube 永久封禁”

## 9. UI 设计建议

## 9.1 页面目标

页面应从“红绿灯式宣判”改为“兼容性画像”。

## 9.2 页面模块

建议展示：

1. 基础连接
2. 默认模式解析
3. 匿名访问能力
4. 身份解锁价值
5. 媒体链路风险
6. 综合结论

## 9.3 文案原则

禁止使用过于绝对的表述：

- 当前节点已被拦截
- 当前节点已被封禁
- 当前 IP 已触发机器人检测

改为：

- 检测到匿名访问受限
- 当前默认模式解析失败
- 媒体链路存在高风险
- 当前登录态可用，但节点兼容性有限

## 9.4 推荐文案示例

### 示例 A

- 综合结论：当前节点可用
- 说明：默认模式可正常解析目标视频，当前代理对 FluentYTDL 主下载路径兼容良好。

### 示例 B

- 综合结论：当前节点有限可用
- 说明：匿名访问受限，但在当前 Cookie 身份下仍可正常解析。

### 示例 C

- 综合结论：当前节点高风险
- 说明：默认模式解析或媒体链路验证失败，建议切换代理节点后重试。

## 10. 与现有实现的关系

## 10.1 保留

1. 代理配置读取逻辑
2. `YoutubeService.build_ydl_options()`
3. 连接池与使用历史机制
4. 节流与缓存框架

## 10.2 弱化

1. `ip_probe.py` 的 `youtubei/player` 固定探针
2. 基于单个 `403/Bot` 的强结论

## 10.3 替换

`network_test_page.py` 应逐步迁移为新的节点兼容性诊断页，底层改为调用统一的 `node_diagnosis.py`。

## 11. 实施分期

## Phase 1：诊断模型纠偏

目标：

1. 去掉 `403/Bot` 的硬判文案
2. 引入默认模式轻量解析检测
3. 将现有测试页改为展示分层结果

交付：

1. `node_diagnosis.py` 初版
2. `NetworkTestPage` 改版

## Phase 2：完善目标相关诊断

目标：

1. 支持优先针对当前 URL 诊断
2. 支持匿名 / 认证态对比
3. 引入风险等级和建议排序

交付：

1. URL 优先级策略
2. 诊断缓存与历史

## Phase 3：深度链路诊断

目标：

1. 增加媒体链路短探测
2. 支持“中途 403”稳定性判断
3. 为节点筛选提供更高质量数据

交付：

1. 深度诊断入口
2. 媒体链路风险分析

---

## 12. 最终结论

本次重构的根本变化是：

从过去的问题：

> “用独立探针判断节点是不是被 YouTube 封了”

转向新的问题定义：

> “当前代理节点在 FluentYTDL 默认模式下，对当前目标视频的兼容性和风险等级如何”

这是一个更符合现状、误报更低、也更能真正帮助用户选节点的方向。

在当前阶段，系统的主判断依据应是：

1. **默认模式是否可解析**
2. **默认模式是否可稳定进入媒体获取阶段**
3. **Cookie 是否仅提供身份解锁，而非被误解释为节点修复手段**
4. **PoToken 仅保留为未来能力，不进入当前主诊断结论**
