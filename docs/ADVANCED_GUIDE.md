## 🧠 1. YouTube 客户端策略与画质原理

### YouTube 的多客户端体系

YouTube 并不是只有一个"服务器"——它根据请求来源的**客户端类型**返回完全不同的视频格式列表。了解这一点是理解画质问题的关键。

| 客户端 | 身份标识 | 特点 |
|--------|----------|------|
| **Web** | 桌面浏览器 | 格式最全（含 4K/8K），但受 SABR 限制（见下文），需要 Cookie + JS 签名 |
| **mweb** | 移动端网页 | PO Token 完整支持，不受 SABR 限制，无 Cookie 时的最佳选择 |
| **android** | Android 应用 | 不支持 PO Token，使用可能触发机器人检测 |
| **android_vr** | Daydream VR | 能获取 VR 专属高分辨率格式（最高 8K），但不支持 Cookie |
| **ios** | iOS 应用 | 部分格式受限 |
| **tv** | 智能电视 | 格式有限 |

### FluentYTDL 的智能客户端策略

FluentYTDL 根据当前 Cookie 状态自动切换客户端组合，这是画质优化的核心逻辑：

**有 Cookie 时 → `web,mweb` 双客户端**
```
Web (优先)  → 获取 4K/8K/Premium 高码率格式
  ↓ 兜底
mweb        → 规避 SABR 限制，确保格式列表完整
```

**无 Cookie 时 → 仅 `mweb`**
```
mweb only   → PO Token 最佳客户端，不受 SABR 影响
```

> [!IMPORTANT]
> **为什么不用 `default`？** `default` 模式可能自动回退到 `android_vr`，而 `android_vr` **不支持 PO Token**，会直接触发 YouTube 的机器人检测。FluentYTDL 故意屏蔽了这个回退路径。

### 什么是 SABR？为什么它影响画质？

**SABR（Server-Adaptive Bitrate Rendering）** 是 YouTube 正在逐步推行的新策略：

- 传统模式：YouTube 返回**完整的格式列表**（从 144p 到 8K 全部列出），客户端自由选择
- SABR 模式：YouTube **仅返回它认为客户端需要的格式**（可能只有 360p/720p），动态调整码率

**影响**：当 Web 客户端被 SABR 限制时，你可能只能看到 1080p 甚至更低的选项。

**FluentYTDL 的应对**：通过 `mweb` 客户端兜底——mweb 目前不受 SABR 影响，始终返回完整格式列表。这也是为什么即使没有 Cookie，FluentYTDL 通常也能获取到不错的画质。

### JS Runtime 与签名解密

YouTube 使用 JavaScript 对视频流 URL 进行**动态签名加密**（称为 cipher/signature）。要获取真正可用的下载链接，必须执行这段 JS 代码来"解密"。

- **有 JS Runtime（Deno/Node.js）**：yt-dlp 能正确解密签名，获取全部格式，并绕过服务器端限速（throttling）
- **无 JS Runtime**：大量高清格式丢失，且下载速度可能被严重限速

> [!WARNING]
> yt-dlp 已正式声明：**没有外部 JS 运行时的 YouTube 支持已被弃用（deprecated）**。请确保 Deno 已安装（`设置 > 核心组件` 检查）。

## 🛡️ 2. PO Token 与反机器人检测

### 什么是 PO Token？

**PO Token（Proof of Origin Token）** 是 YouTube 用于验证请求来源合法性的数字令牌。它的核心作用是向 YouTube 证明："这个请求确实来自一个真实的浏览器/客户端，而不是自动化脚本。"

没有 PO Token 时可能发生：
- 触发 "Sign in to confirm you're not a bot" 人机验证
- 高画质格式被隐藏（仅返回 720p 或更低）
- 直接返回 HTTP 403 拒绝访问

### bgutil-ytdlp-pot-provider 的工作原理

FluentYTDL 集成了 **bgutil-ytdlp-pot-provider** 来自动生成 PO Token，流程如下：

```
1. 应用启动时，POT Provider 在本地启动一个 HTTP 服务（默认端口 4416）
2. 服务内部使用 Deno 运行时加载 Google 的 BotGuard 挑战脚本
3. 每次 yt-dlp 需要解析视频时，向 POT Provider 请求一个新的 Token
4. POT Provider 执行 BotGuard 挑战，生成 Token 并返回
5. yt-dlp 将 Token 通过 extractor_args 附加到请求中
6. YouTube 验证 Token 有效性，正常返回视频数据
```

### PO Token 与 Cookie 的关系

它们是**互补**关系，而非替代：

| | PO Token | Cookie |
|--|----------|--------|
| **证明什么** | 请求来自合法客户端 | 用户身份和权限 |
| **解决什么** | 机器人检测、基础格式访问 | 年龄限制、会员内容、高码率流 |
| **缺失后果** | 触发人机验证、画质受限 | 无法下载受限内容 |

**最佳实践**：两者都启用。PO Token 确保请求不被拦截，Cookie 确保有权限获取完整内容。

### POT Provider 的预热机制

POT Provider 首次启动时需要初始化 BotGuard 环境（下载挑战脚本、编译 WASM 模块等），这个过程称为 **warm-up（预热）**。

- 预热通常需要 5-15 秒
- 预热期间 yt-dlp 的解析可能会暂时缺少 PO Token
- FluentYTDL 会在日志中显示 `⏳ POT Provider 正在初始化，请稍候...`
- 预热完成后显示 `🛡️ POT Provider 已激活: 端口 4416`

> [!TIP]
> 如果 POT Provider 一直无法激活，请检查 Deno (JS Runtime) 是否已正确安装。POT Provider 依赖 Deno 来执行 BotGuard 脚本。

## 🍪 3. Cookie 全流程解析

### DLE 登录获取 — 技术原理

DLE 的全称是 **Dynamic Local Extension**（动态本地扩展），但当前版本已演进为基于 **WebView2** 的实现方案。

**架构**：双进程模型
```
主进程 (Qt GUI)  ──→  multiprocessing.Process  ──→  子进程 (pywebview + WebView2)
                 ←── multiprocessing.Queue      ←── Cookie 轮询检测
```

**完整流程**：

1. **启动安全沙盒**：通过 `pywebview` 创建一个使用 Edge WebView2 引擎的独立浏览器窗口
   - 使用独立的 `storage_path`（缓存目录），与您的主浏览器完全隔离
   - `private_mode=False`：保持登录状态的持久化，支持下次免登录

2. **用户登录**：窗口加载 `youtube.com`，用户正常登录 Google 账号

3. **Cookie 轮询检测**：后台线程每 2.5 秒检查一次
   - 检测 YouTube 域是否出现 `LOGIN_INFO` Cookie（登录完成标志）
   - 检测到后，自动导航到 `accounts.google.com` 采集 Google 域 Cookie
   - 合并两个域的 Cookie，进行去重和格式化

4. **合规清洗与回传**：
   - `CookieCleaner` 过滤掉非必要的 Cookie（只保留 YouTube/Google 认证所需）
   - 转换为 Netscape 格式写入缓存
   - 通过 `multiprocessing.Queue` 安全传递回主进程
   - Cookie Sentinel 自动同步到 `bin/cookies.txt` 供 yt-dlp 使用

5. **清理**：子进程自动销毁

**多账号管理**：每个账号使用独立的 WebView2 profile 目录。在设置页可添加、切换、删除账号。已登录的账号可跳过登录直接提取最新 Cookie。

### 自动浏览器提取 — 兼容性深度解析

FluentYTDL 使用 `rookiepy` 库从本地浏览器的 Cookie 数据库文件中直接读取数据。

**为什么 Firefox 最好？**

Firefox 使用 SQLite 存储 Cookie，加密方式简单（NSS），不依赖操作系统级的 DPAPI。任何程序都能在用户权限下直接读取，无需额外权限。

**为什么 Chrome 不行？**

自 Chrome 127 版本起，Google 引入了 **App-Bound Encryption（应用绑定加密）**：

```
Chrome v126 及更早    Chrome v127+
─────────────────    ──────────────────────
Cookie DB             Cookie DB
  ↓                     ↓
DPAPI 加密             App-Bound Encryption
(管理员可解)            (绑定到 Chrome 进程)
  ↓                     ↓
rookiepy 可读          ❌ 第三方无法解密
```

- **Edge**：同样受 App-Bound 影响，但部分版本在管理员权限下仍可通过 DPAPI 解密 → 需以管理员身份运行 FluentYTDL
- **Chrome**：即使管理员权限也可能返回 `decrypt_encrypted_value failed` → 基本不可用
- **其他 Chromium 浏览器**（Brave、Opera、Vivaldi 等）：行为因版本而异，大多需要管理员权限

### Cookie Sentinel 同步机制

无论 Cookie 从哪种来源获取，最终都要经过 **Cookie Sentinel** 统一同步到 `bin/cookies.txt`，这是 yt-dlp 实际读取的唯一文件。

```
DLE 登录获取  ──┐
                 │
Firefox 提取  ──┼──→ CookieCleaner 合规清洗 ──→ bin/cookies.txt ──→ yt-dlp --cookies
                 │
手动导入 .txt ──┘
```

**CookieCleaner 的清洗规则**：
- 只保留 `.youtube.com` 和 `.google.com` 域的 Cookie
- 白名单制：保留登录必需的核心 Cookie（SID、HSID、SSID、SAPISID、APISID 等）
- 过滤已过期的 Cookie
- 清除可能造成风控的非必要追踪 Cookie

### Cookie 探测节流（Probe Throttle）

FluentYTDL 内置了智能的 Cookie 有效性探测机制，但为了**避免探测本身触发 YouTube 风控**，实施了严格的节流策略：

- **日限**：每天最多探测若干次
- **间隔**：两次探测之间保持最短间隔
- **风控退避**：连续失败后自动延长间隔（指数退避）
- **手动模式**：用户手动触发的探测有独立的最短间隔

探测逻辑采用**交叉验证**：先带 Cookie 解析 → 失败后不带 Cookie 解析 → 对比结果判断是 Cookie 问题还是 IP 问题。

## 🎮 4. VR 视频技术详解

### VR 视频检测机制

FluentYTDL 解析视频时会自动检测是否为 VR 内容，触发条件（满足任一即可）：

1. **关键词检测**：标题或描述中包含 VR180、VR360、360°、SBS、side by side、stereoscopic、immersive 等关键词
2. **格式元数据检测**：视频格式中包含 `mesh` 标记（VR 投影格式特征）
3. **分辨率异常检测**：标题声称 8K 但格式列表最高低于 4320p — 这是 VR 视频的典型特征（普通客户端获取不到完整分辨率）

检测到 VR 视频后，FluentYTDL 会自动使用 `android_vr` 客户端进行二次解析，获取 VR 专属的高分辨率格式并合并到格式列表中。

### 三种投影类型

VR 视频在 2D 屏幕上显示为"畸变"画面，是因为它们使用了特殊的投影方式将球面画面映射到平面：

| 投影类型 | 全称 | 特点 | 画面特征 |
|----------|------|------|----------|
| **Equirectangular** | 等距柱状投影 | 最常见的 360° 格式，宽高比 2:1 | 上下拉伸、极点变形 |
| **Mesh** | 鱼眼/网格投影 | VR180 常用，YouTube 高端编码 | 圆形鱼眼画面 |
| **EAC** | 等角立方体投影 | YouTube 自研的高效率格式 | 六面展开图，非标准比例 |

FluentYTDL 会自动分析每个视频格式的投影类型，判断依据：
- `mesh` 标记 → Mesh 投影
- 宽高比 2:1 或 1:1 → Equirectangular
- 宽高比 ~1.5:1 + 非 H.264 编码 → EAC

### 立体模式

| 模式 | 说明 | 常见来源 |
|------|------|----------|
| **Mono（2D 全景）** | 单眼画面，360° 环顾 | 大部分 360° 视频 |
| **Stereo SBS（左右 3D）** | 左右各半张画面，VR 头显合成 3D | VR180 视频、Mesh 投影 |
| **Stereo TB（上下 3D）** | 上下各半张画面 | 1:1 宽高比的 360° 3D 视频 |

### EAC 转码注意事项

> [!WARNING]
> EAC 格式在本地播放器兼容性较差，通常需要 FFmpeg 转码为 Equirectangular 格式。**8K EAC 转码极度消耗资源**：
> - 建议配备 **32GB+ 内存** 和高端显卡
> - 8K 转码可能需要**数小时**
> - 没有高性能硬件时，建议将 VR 最大分辨率限制在 4K/6K（`设置 > VR 设置`）

## ⚡ 5. 性能调优与并发控制

### 并发分片数

FluentYTDL 使用多线程分片下载来加速。在 `设置 > 下载选项` 中可以调整 `concurrent_fragments`（并发分片数）。

| 设置值 | 适用场景 | 风险 |
|--------|----------|------|
| 1-2 | 网络不稳定、代理带宽小 | 无 |
| **4**（默认） | 大多数场景的最佳平衡 | 无 |
| 8-16 | 高带宽直连、需要极速下载 | 可能触发 429 限流 |
| 16+ | 不推荐 | 高概率限流、IP 封锁 |

> [!TIP]
> 分片数并非越大越快。YouTube 会对单 IP 的并发连接数进行监控，超过阈值会触发 429 限流甚至暂时封锁 IP。**默认值 4 是经过大量测试的最佳平衡点。**

### 下载限速

在 `设置 > 下载选项` 中可设置下载速率上限（rate_limit），适用场景：
- 避免下载占满全部带宽影响其他应用
- 降低被 YouTube 风控检测的概率
- 后台长时间挂机下载时

### SponsorBlock 广告跳过

FluentYTDL 集成了 SponsorBlock，可自动移除或标记 YouTube 视频中的赞助广告片段。

**工作原理**：
1. SponsorBlock 社区维护了一个**众包标注数据库**，记录了数百万视频中的广告时间段
2. yt-dlp 下载视频时向 SponsorBlock API 查询当前视频的广告标注
3. 使用 FFmpeg 自动**裁剪**（remove）或**标记为章节**（mark）

**支持的类别**（可在设置中自由选择）：
- `sponsor`：赞助广告
- `selfpromo`：频道自我推广
- `interaction`：订阅提醒、点赞提醒等

在 `设置 > 下载选项 > SponsorBlock` 中启用并选择类别和处理方式。

### 批量播放列表下载最佳实践

1. **合理并发**：不要同时启动过多下载任务。FluentYTDL 内置任务队列管理，建议保持默认的并发任务数
2. **全局格式预设**：在播放列表工具栏使用"格式设置"按钮统一配置分辨率和格式，而非逐个视频设置
3. **避免风控**：
   - 一次性解析超大播放列表（500+ 视频）时，YouTube 更容易触发风控
   - 建议分批解析，或在解析间隔适当等待
   - 确保 Cookie 和 PO Token 都已启用

## 🔧 6. 组件体系与更新渠道

### 组件安装位置

| 环境 | 安装路径 |
|------|----------|
| 打包发布版 | `{安装目录}/_internal/bin/{组件名}/` |
| 开发模式 | `{项目根目录}/assets/bin/{组件名}/` |

每个组件独立一个子目录，包含可执行文件和可能的配置文件。

### yt-dlp 的三个更新渠道

yt-dlp 是 FluentYTDL 最核心的组件，更新频率最高。提供三个渠道：

| 渠道 | 更新频率 | 稳定性 | 适用场景 |
|------|----------|--------|----------|
| **stable** | 数周一次 | ⭐⭐⭐ 最稳定 | 日常使用（默认推荐） |
| **nightly** | 每日自动构建 | ⭐⭐ 较稳定 | YouTube 大改版后 stable 还没修复时 |
| **master** | 每次代码合并 | ⭐ 可能不稳定 | 开发测试用途 |

**切换方法**：`设置 > 核心组件 > yt-dlp` 中可选择渠道。

> [!TIP]
> **紧急自救指南**：当 YouTube 突然大规模改版导致所有下载失败时：
> 1. 首先尝试更新 stable 渠道 — 大部分情况下社区会快速修复
> 2. 如果 stable 还没跟上，临时切换到 **nightly** — 包含最新的修复补丁
> 3. 修复被合并到 stable 后再切回来

### 镜像源配置

组件更新需要从 GitHub 下载。中国大陆用户可能遇到下载缓慢或失败的问题。

在 `设置 > 核心组件` 中可切换下载源：
- **GitHub 直连**（默认）：速度取决于您的网络环境
- **ghproxy 镜像**：通过代理加速 GitHub 下载

> [!NOTE]
> 组件更新也会使用您在 `设置 > 网络连接` 中配置的代理。如果更新失败，请确认代理能访问 `api.github.com`。

### 版本检测逻辑

FluentYTDL 的版本比较采用**语义化版本元组**比较：
- 将版本字符串（如 `2026.04.15`）解析为整数元组 `(2026, 4, 15)`
- 短版本自动补零对齐：`7.1` vs `7.1.3` → `(7, 1, 0)` vs `(7, 1, 3)`
- 仅当远程版本**严格大于**本地版本时才提示更新

**跨渠道切换**：当您从 stable 切换到 nightly（或反之），无论版本号大小，都会**强制触发更新**以确保渠道一致性。

## 📦 7. 视频格式、编码与容器兼容性

这是用户最常困惑的领域之一："为什么下载的视频在手机上/电视上/某个播放器里无法播放？" 要理解这个问题，需要区分三个概念：**编解码器（Codec）**、**容器格式（Container）** 和 **封装兼容性**。

### 三个核心概念

```
视频文件 = 容器（包装盒）+ 视频编码（画面数据）+ 音频编码（声音数据）+ 字幕/元数据

类比：
  容器  = 快递箱（MP4、MKV、WebM）
  视频编码 = 箱子里的大件商品（H.264、VP9、AV1）
  音频编码 = 箱子里的小件商品（AAC、Opus、Vorbis）
```

问题出在：**不是所有箱子都能装所有商品**，**不是所有收件人（播放器）都能打开所有箱子**。

### YouTube 常见视频编码

| 编码 | 技术名 | 特点 | 兼容性 |
|------|--------|------|--------|
| **H.264** | AVC (avc1) | 最老牌、兼容性最强，几乎所有设备都能播放 | ⭐⭐⭐ 全平台 |
| **VP9** | vp9/vp09 | Google 主推的开源编码，YouTube 高画质流主力 | ⭐⭐ PC/手机可以，部分电视不行 |
| **AV1** | av01 | 最新一代开源编码，压缩效率最高（同画质体积最小） | ⭐ 新设备支持，老设备/电视大多不行 |

**YouTube 的编码策略**：
- 1080p 及以下：通常同时提供 H.264 和 VP9
- 1440p/4K：以 VP9 为主，部分提供 AV1
- 4K HDR / 8K：几乎只有 VP9 或 AV1

这意味着：**画质越高，编码越新，兼容性越差。** 这是技术发展的必然。

### YouTube 常见音频编码

| 编码 | 格式扩展名 | 特点 | 兼容性 |
|------|-----------|------|--------|
| **AAC** | .m4a | 苹果和 MPEG 阵营的标准，兼容性极佳 | ⭐⭐⭐ 全平台 |
| **Opus** | .webm | Google 主推的开源音频编码，音质/压缩比更优 | ⭐⭐ 大部分现代设备 |
| **Vorbis** | .webm | 较老的开源音频编码，正在被 Opus 取代 | ⭐⭐ 大部分设备 |

**重点**：YouTube 的高码率音频（256kbps/384kbps Premium）通常使用 **Opus** 编码。如果你追求最佳音质，大概率会遇到 Opus。

### 三种容器格式

| 容器 | 扩展名 | 原生支持的编码 | 字幕支持 | 多音轨 |
|------|--------|---------------|----------|--------|
| **MP4** | .mp4 | H.264 + AAC ✅ / VP9 ⚠️ / Opus ❌ | 单轨可用（mov_text） | 有限 |
| **MKV** | .mkv | 几乎所有编码 ✅ | 完整支持（多轨、多格式） | 完整支持 |
| **WebM** | .webm | VP9/AV1 + Opus/Vorbis ✅ / H.264 ❌ | 不支持 SRT/ASS | 有限 |

### 为什么有些视频下载后无法播放？

理解了上面的基础后，以下常见问题的原因就清晰了：

#### 场景 1：MP4 文件在电视/手机上无声音

**原因**：视频选了 VP9/AV1 编码 + Opus 音频，然后封装到了 MP4 容器中。

```
❌ 问题组合：MP4 容器 + VP9 视频 + Opus 音频
   → Opus 无法原生封入 MP4，FFmpeg 被迫将 Opus 转码为 AAC
   → 转码后码率/质量可能下降
   → 部分播放器甚至直接无法解析
```

**FluentYTDL 的应对**：格式评分引擎会在目标容器为 MP4 时，**优先选择 AAC (m4a) 音频流**而非 Opus，即使 Opus 的码率略高。这确保了无损封装（remux）而非转码。

#### 场景 2：4K 视频在老旧设备上只有声音没有画面

**原因**：4K 流几乎只有 VP9/AV1 编码，老旧设备（如 2018 年前的电视、老款手机）的硬件解码器不支持这些编码。

**解决方案**：
- 选择 1080p 以下的 H.264 流（兼容性最佳）
- 或使用支持软解的播放器（如 VLC、PotPlayer、MX Player）

#### 场景 3：嵌入字幕后文件变成了 MKV

**原因**：FluentYTDL 检测到字幕兼容性冲突后自动升级了容器。

```
用户选择：MP4 + 嵌入 3 种语言字幕
   → MP4 的 mov_text 字幕格式对多语言支持差（部分播放器只显示第一条）
   → FluentYTDL 自动升级为 MKV（完整支持多轨字幕切换）
```

同理，如果选择了**多音轨嵌入**（如日语+中文配音），也会自动升级为 MKV，因为 MP4 在部分播放器中无法正常切换音轨。

#### 场景 4：WebM 文件很多播放器打不开

**原因**：WebM 是 Google 的开放容器格式，Windows 自带播放器和部分手机播放器不原生支持。

**解决方案**：安装 VLC 或 PotPlayer，或选择 MP4/MKV 作为输出容器。

### FluentYTDL 的智能容器决策

FluentYTDL 内置了一套 **容器兼容性引擎**，在下载时自动做出最优决策：

```
决策流程：

1. 多音轨嵌入？        ──→  强制 MKV
2. 多语言字幕嵌入？    ──→  强制 MKV
3. 视频 mp4 + 音频 m4a？ ──→  MP4（无损 remux）
4. 视频 webm + 音频 webm？──→  WebM（无损 remux）
5. 其他混合组合？       ──→  MKV（万能容器，无损 remux）
6. 单字幕 + WebM？      ──→  升级为 MKV（WebM 不支持 SRT）
```

> [!TIP]
> **什么是 remux？** Remux（重封装）是把已编码的音视频数据从一个容器搬到另一个容器，**不重新编码**，因此速度极快（几秒完成）且画质零损失。只有当编码与目标容器不兼容时（如 Opus → MP4），才会触发慢速的重新编码（re-encode）。

### 最佳实践建议

| 你的需求 | 推荐容器 | 理由 |
|----------|----------|------|
| 最大兼容性（手机/电视/微信发送） | **MP4** | H.264+AAC 组合全平台通吃 |
| 最佳画质保真（收藏/归档） | **MKV** | 万能容器，原封不动保留所有编码 |
| 嵌入多语言字幕/多音轨 | **MKV** | MP4/WebM 对多轨支持不佳 |
| 纯音频提取 | **MP3/M4A** | MP3 兼容性最强，M4A 音质更好 |

> [!NOTE]
> FluentYTDL 默认选择 MP4 作为输出容器（简易模式）。当检测到不兼容的编解码器组合时，会在 UI 中显示 ⚠️ 警告提示，并在必要时自动升级为 MKV 以避免播放问题。您也可以在下载配置面板中手动指定容器格式。
